The Doc Optimizer Server requires several AWS services to be configured. This guide walks you through setting up S3, DynamoDB, and Lambda.

## Prerequisites

Before you begin, ensure you have:
- An AWS account
- AWS CLI installed and configured
- Appropriate IAM permissions to create resources

## S3 Bucket Configuration

The server uses S3 to store original uploads and processed documents.

### Create the Bucket

```bash
aws s3 mb s3://doc-optimizer-uploads --region us-east-1
```

### Set Up CORS (Optional)

If you're uploading from a web browser, configure CORS:

```json
[
  {
    "AllowedHeaders": ["*"],
    "AllowedMethods": ["PUT", "GET"],
    "AllowedOrigins": ["https://yourdomain.com"],
    "ExposeHeaders": ["ETag"],
    "MaxAgeSeconds": 3000
  }
]
```

Apply the CORS configuration:

```bash
aws s3api put-bucket-cors --bucket doc-optimizer-uploads --cors-configuration file://cors.json
```

### Folder Structure

The server automatically organizes files using these prefixes:
- `uploads/original/` - Stores uploaded files with UUID-based names
- `uploads/processed/` - Stores optimized files with format: `{jobId}_{optimizationLevel}.pdf`

<Note>
  The server uses pre-signed URLs for secure upload and download operations with 60-second upload expiry and 300-second (5-minute) download expiry (see `src/services/s3Service.js:19,29`).
</Note>

### Lifecycle Policy (Recommended)

Set up a lifecycle policy to automatically delete old files:

```json
{
  "Rules": [
    {
      "Id": "DeleteOldUploads",
      "Status": "Enabled",
      "Prefix": "uploads/",
      "Expiration": {
        "Days": 7
      }
    }
  ]
}
```

## DynamoDB Tables

You need to create three DynamoDB tables for jobs, API keys, and usage tracking.

### Jobs Table

Stores processing job metadata and status.

```bash
aws dynamodb create-table \
  --table-name doc-optimizer-jobs \
  --attribute-definitions AttributeName=jobId,AttributeType=S \
  --key-schema AttributeName=jobId,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST \
  --region us-east-1
```

**Table Schema:**
- `jobId` (String, Partition Key) - Unique job identifier
- `status` (String) - Job status: `CREATED`, `PROCESSING`, `COMPLETED`, `FAILED`
- `inputKey` (String) - S3 key for the original file
- `outputKey` (String) - S3 key for the processed file
- `optimizationLevel` (String) - Requested optimization level
- `createdAt` (String) - ISO timestamp
- `updatedAt` (String) - ISO timestamp

### API Keys Table

Manages API authentication and authorization.

```bash
aws dynamodb create-table \
  --table-name doc-optimizer-api-keys \
  --attribute-definitions AttributeName=apiKey,AttributeType=S \
  --key-schema AttributeName=apiKey,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST \
  --region us-east-1
```

**Table Schema:**
- `apiKey` (String, Partition Key) - The API key
- `clientId` (String) - Client identifier
- `active` (Boolean) - Whether the key is active
- `expiresAt` (String, Optional) - ISO timestamp for expiration

<Note>
  The authentication middleware checks both the `active` status and optional `expiresAt` field (see `src/middleware/auth.js:12,16`).
</Note>

### Usage Tracking Table

Logs API usage for analytics and billing.

```bash
aws dynamodb create-table \
  --table-name doc-optimizer-usage \
  --attribute-definitions \
    AttributeName=usageId,AttributeType=S \
    AttributeName=timestamp,AttributeType=S \
  --key-schema \
    AttributeName=usageId,KeyType=HASH \
    AttributeName=timestamp,KeyType=RANGE \
  --billing-mode PAY_PER_REQUEST \
  --region us-east-1
```

**Table Schema:**
- `usageId` (String, Partition Key) - UUID for the usage record
- `timestamp` (String, Sort Key) - ISO timestamp
- `clientId` (String) - Client who made the request
- `endpoint` (String) - API endpoint called
- `jobId` (String, Optional) - Associated job ID if applicable

<Warning>
  Make sure to create all three tables in the same AWS region as your S3 bucket and Lambda function.
</Warning>

## Lambda Function Setup

The processing Lambda function is invoked asynchronously to handle document optimization.

### IAM Role

Create an IAM role for the Lambda function with these permissions:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": "arn:aws:s3:::doc-optimizer-uploads/*"
    },
    {
      "Effect": "Allow",
      "Action": [
        "dynamodb:UpdateItem",
        "dynamodb:GetItem"
      ],
      "Resource": "arn:aws:dynamodb:us-east-1:*:table/doc-optimizer-jobs"
    },
    {
      "Effect": "Allow",
      "Action": [
        "logs:CreateLogGroup",
        "logs:CreateLogStream",
        "logs:PutLogEvents"
      ],
      "Resource": "arn:aws:logs:*:*:*"
    }
  ]
}
```

### Lambda Configuration

<Note>
  The server invokes the Lambda function with `InvocationType: 'Event'` for asynchronous execution (see `src/services/lambdaService.js:12`).
</Note>

Recommended settings:
- **Runtime:** Node.js 18.x or later
- **Memory:** 512 MB - 2 GB (depending on document size)
- **Timeout:** 5-15 minutes (depending on processing complexity)
- **Environment Variables:** Same as the server (AWS_REGION, S3_BUCKET, DYNAMO_TABLE)

The API server (when deployed as Lambda) needs these permissions:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:PutObject",
        "s3:GetObject"
      ],
      "Resource": "arn:aws:s3:::doc-optimizer-uploads/*"
    },
    {
      "Effect": "Allow",
      "Action": [
        "dynamodb:PutItem",

        "dynamodb:UpdateItem"
      ],
      "Resource": [
        "arn:aws:dynamodb:us-east-1:*:table/doc-optimizer-jobs",
        "arn:aws:dynamodb:us-east-1:*:table/doc-optimizer-api-keys",

      "Action": "lambda:InvokeFunction",

    }
  ]
}
```

## Testing Your Setup

After configuring all services, verify the setup:

### 1. Check S3 Access

```bash
aws s3 ls s3://doc-optimizer-uploads/
```

### 2. Verify DynamoDB Tables

```bash
aws dynamodb list-tables --region us-east-1
```

### 3. Test Lambda Invocation

```bash
aws lambda invoke \
  --function-name doc-optimizer-processor \
  --invocation-type Event \
  --payload '{"test": true}' \
  response.json
```

### 4. Create a Test API Key

```bash
aws dynamodb put-item \
  --table-name doc-optimizer-api-keys \
  --item '{
    "apiKey": {"S": "test-key-12345"},
    "clientId": {"S": "test-client"},
    "active": {"BOOL": true}
  }'
```

## Cost Optimization

<Note>
  Using `PAY_PER_REQUEST` billing mode for DynamoDB is recommended for variable workloads. Switch to provisioned capacity if you have predictable traffic patterns.
</Note>

Consider these strategies:
- **S3 Lifecycle Policies:** Automatically delete old files to reduce storage costs
- **DynamoDB On-Demand:** Use PAY_PER_REQUEST for unpredictable traffic
- **Lambda Reserved Concurrency:** Set limits to control costs
- **CloudWatch Alarms:** Monitor usage and set billing alerts
